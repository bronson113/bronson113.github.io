### Vault
```
authored by v3ct0r, Chainfire73

Can you perform side-channel attack to this vault? The PIN is a 10-digit number.

* Python3 is installed on remote. `nc vault.chal.2024.ctf.acsc.asia 9999`
```
We are given the binary file run on the remote server. With a bit of reversing the challenge, we see that it first checked the length of the pin, then check the pin 1 by 1. Each check adds a 0.1 second delay, so the timing difference is very observable. A simple timing side channel attack can be used to extract the pin. After the pin is extracted, running the binary with the pin gives the flag.

`ACSC{b377er_d3L4y3d_7h4n_N3v3r_b42fd3d840948f3e}`

From the source, we know that the strongest_random function can only return 16 different possible values, and the mask is always constant. 
Now if we look at how OAEP padding works, this means that for the same message, only 16 possible padded message exists, and we can calculate the difference between each of them quite easily. 
Since we are given two sets of encrypted values that shares the modulo, we can apply some form of related message attack. Namely, the [FranklinReiter Attack](https://github.com/ValarDragon/CTF-Crypto/blob/master/RSA/FranklinReiter.sage).

The idea is that if we know how the two messages are related, lets say it's $x$ and $f(x)$, then the following two equations will hold when $x = m$.

$$\begin{align} 
x^{e_1} - c_1 &= 0 &\mod{n} \\  
f(x)^{e_2} - c_2 &= 0 &\mod{n} \\  
\end{align}$$

which means that both equation contains the factor $(x-m)$. We can now apply GCD on the two equation to recover the m.

In this challenge, $f(x)$ will simply be $x+k$ where k is the difference in the random value. We don't know the exact difference value though, so we iterate all possible difference in hope to find the value we need. In my script I add a special case where the difference is 0, which can be solved in an easier method. That end up not being the case, but I leave it in the script.

The website hosts a simple login page. If we look at the login route, our username is first used to retrieve the user object, then our password is checked. Lastly, it check if our username is guest and prevent us from viewing the flag if that's the case. Notice that `username === 'guest'` is a strict comparison while all other checks are loss. This means that if username is somehow an array, this comparison will always fail. On the other hand, `const user = USER_DB[username];` will retrieve the user even when username is an object. It'll try to coerce the object into string and use that as index instead. We abuse this to login as guest but still retrieve the flag.

During the competition, I'm completely clueless as to what those instructions does, I looked up `MATCH_SEQUENCE` instruction, but there doesn't seems to be any sequence in this file, and I thought that the tuple isn't a sequence, so it would push a false onto the stack. And all the other arithmatics just seems weird to me. 

After the competition ends, I looked up how other people approach this challenge. 
There are mainly two approachs, either print out the value after it's calculated by the file itself, or just understand how the values are constructed. 
For the first approach, if we look at the disassembled bytecode, we see that in the end, our value is compared using the COMPARE_OP. 
If at that point we instead print out that value, using `input` for example, we can straight up print out the flag. Patching run.pyc file using a hex editor or anything of your choice can achieve this. 

Now for understanding the program. It turns out the MATCH_SEQUENCE will match on tuple, and push a true onto the stack, which is just a 1 for arithmatic purposes. If we look at the 4 instruction sequence
```
        2      BUILD_TUPLE                   0
        4      MATCH_SEQUENCE
        6      ROT_TWO
        8      POP_TOP
```
It pushs an empty tuple, then pushs a true, and pop the empty tuple. Therefore, the whole sequence is simply just pushing a 1 onto the stack. Now the other operations strings together to constructing all the values needed for the comparison. On byte 2412, all the values on the stack is turned to a tuple, called with bytes and str (found through experiements and printing out values with `input`). The tuple is then compared with the user input. Simulating this procedure is easy enough, just parse the disassemble output a little bit, and print out the final stack gives us the flag.
